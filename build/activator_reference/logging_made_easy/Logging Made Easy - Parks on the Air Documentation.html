    POTA 的日志处理为什么比较复杂？

    这个问题无论ham老鸟还是新手，在刚玩儿 POTA 时经常会问。他们习惯于就是简单地把呼号往日志软件或网站里一输，就搞定。

    好吧，答案是：POTA 的日志处理其实也没多复杂。只是有些简单参数（或说规则）用来确保激活人/猎人双方都能赢取相关的积分而已。

    是的，POTA 是一项积分驱动的奖项活动，其网站、docs.pota.app 等上面说明了相关规则。但本文是关于日志这块儿的，好我们继续...

    POTA 使用“业余无线电数交换格式”（ADIF）文件处理日志，这是种日志软件和日志数据库如 QRZ、eQSL、LoTW等之间进行数据同步的有效格式，在ham届广泛应用。

    为了保持一致性方便性，ham常用的任何日志软件所生成的 ADIF 格式文件都可被 POTA 平台所兼容。

    ADIF 文件是怎么工作的？

    想象一个含有行列的数据表程序，例如 Excel。

    ADIF 里，每列称为“字段”，每行就是你的通联数据。各类日志软件读取 ADIF 文件后会寻找表头所定义的特定字段，然后将其提取出来用于处理分析。

    你的激活人日志、猎人记录、激活状态、奖项等由此得出。

    POTA 日志里须要有些什么？

    POTA 只有5个必填字段，对某些特定台站另有5个额外字段。这些字段被称之为“标签”（Tag）。

    五个必填字段是：CALL (对方呼号) QSO_DATE, TIME_ON, MODE 及 BAND。这都无须解释。没有更多必须的字段了，虽说多放一些字段也没问题。

    单人单公园的激活这些字段就够了。只是在从纸质日志誊抄时记得将日期换算成正确的 UTC 时间！

    额外的那些字段是干什么用的？

    第一组“可选”字段是用于公园对公园（P2P）的。有些人可能会告诉你用不到这些啦、系统会自动缓存他们啦... 之类。虽说可能是这样的，但多做点互相参照多做点保险总没错。

    这些组字段是 MY_SIG_INFO 及 SIG_INFO。 “SIG” 意思是特殊兴趣组（Special Interest Group）。 MY_SIG_INFO 里会以位置指示字母 (K, JA, VK, PA)+短横杠（-）+ 四个数字的格式放置你的公园ID。 无论是否是 P2P，对每个通联来说这个字段都是一样的。如果你不是在 P2P，这个字段会被忽略；但在 P2P 情形下，你将通联对方的公园ID记录在 SIG_INFO 字段里，然后这两个字段一起就能确保你获取相应的 P2P 积分（当然对方也要以有效格式上传日志才行）。

    注意：如果你通联了一个买一送n的激活人，记得在你日志文件里增加对应条目，并分别填好 SIG_INFO 以防被去重。

    所有数据（甚至时间）都可以相同，但公园号码不能一样。这就是为什么公园ID在我们系统里是唯一的。

    不同操作员共用一个呼号（俱乐部激活）

    下述字段因为不常用经常会被漏下，它们是STATION_CALLSIGN 及 OPERATOR。在同一个台有多个关联呼叫时这些字段会被用到。

    例如：Galifray 的“我爱无线电”俱乐部以 W1WHO 的呼号在他们的周年纪念活动里激活了一个POTA 公园，俱乐部成员 N1ABC 负责话筒/按键。俱乐部和操作员都要在 POTA 里建有注册账户。

    因为电波里使用的是俱乐部呼号，这个呼号要填入 STATION_CALLSIGN 字段；操作员的呼号要填入 OPERATOR 字段。这些字段在设置日志软件的时候就配好，然后会一直持续带入各条通联记录内。

    提交日志之后，俱乐部和操作员都会分别赢得积分，但与他们通联的猎人只会赢得一次积分，因为这是一次通联。如果操作员单独做了呼叫，那么他/她需要单独上传自己的日志。

    当操作员有轮值时，必须记得在日志文件里切换 OPERATOR 字段。

    一个公园跨多个地理位置

    如果你激活的公园横跨多个州/省市/区县等，你应当将特殊的地址记录在 MY_STATE 字段中。这会使得我们的系统得以为猎人 WAS 奖项、为打造自己的州激活矩阵的激活者们分配相应的积分。有很多的公园、历史路径、河流等横跨多个州/省市/区县，因此将它们放入你的文件中就是你自己的职责。如果你没有将其放入 MY_STATE 字段，那你就需要在上传文件时手动指定。

    MY_STATE 字段会在美国、加拿大、澳大利亚、俄罗斯的相关位置解析时生效。

    多个操作员（非俱乐部）怎么处理？

    当两个以上操作员在同一个公园激活时，假设他们都做了相同的通联，那么可以复制出同一个日志并将其根据各自的呼号分别命名（后面会说到名字）。

    如果正在某公园里操作一个特别事件电台（SES）呢？

    对于 SES 呼叫，如果是单一一个操作员，那他须要在他的主账户下创建第二个账号，并将时间起止根据事件活动的要求设好（如果使用了1x1短呼号，记住这些呼号可能日后会被回收重复利用，因此设置好相应的起止时间对赢取积分很重要）。事件呼号记录在 Station_Callsign 字段里，然后操作员呼号记录在 Operator 字段里。猎人们仅会根据电波里传来的 SES 呼叫赢取积分，而操作员会赢取相关的激活积分。

    如果 SES 激活对应着多名操作员，那么必须使用俱乐部账户，这样各个操作员才能获得他们各自的激活积分。俱乐部的日志文件处理不同于个体的日志文件处理。对于个体来说，Operator 字段不会被识别，如有会被拒掉，因为个体账户无法关联其他操作员。

    SES 的多操作员打点

    如果在我们的打点页面上由多人对同一个呼号进行打点，无论是否在同一个波段/模式，必须在呼号后面以 /X（唯一标识符）分别进行后缀区分，这样会避免前一个呼号将后面的同一个呼号冲掉。但电波里呼叫时不要带有后缀。建议采用波段进行后缀，例如 K3Y/40 表示40米上的呼叫。

    有些人会在自己的日志里加上 /X 的后缀，但这只用在 POTA 之外。更重要的事情是确保你的 P2P 方仅记录不带后缀的 SES 呼号，你的事件活动日志里也一样。猎人无论日志与否都会获得应有积分。

    如何记录多公园激活以赢取全部积分？

    在买一送n情形里，日志也没多复杂。记得在初始化阶段将激活的公园号码填入 MY_SIG_INFO 字段，后继过程中它自然会带进各条通联记录里。期间做的 P2P 通联，将对应的公园ID填入 SIG_INFO 字段。

    请勿 使用其他字段名！我们已经见过太多人为 P2P 创建他们自己的字段标签，然后跑来帮助台询问为什么他们的条目匹配不出来。我们系统唯一识别的 P2P 字段就是上面描述的那些，请勿自造。

    完成之后，可以用任意文本编辑器，例如 Windows 自带的记事本，或者 ADIF Master等任意能读取/编辑 ADIF 文件的软件打开日志文件。如果用的是 ADIF Master，找到 MY_SIG_INFO 字段，然后选中该列所有单元格，点击右键，然后选择 Replace in Selection。在查找框里它应该会将公园号码显示出来，然后你在替换框里可以输入第二个第三个公园号码，应用更新后点击 FILE 下的 Save As，再以新的公园号码将文件重命名后存好（假设你已经将原文件以合适的名字存好了）。

    如果用每台 Windows 自带的记事本程序，则用其打开文件，找到编辑菜单里的替换，查找框中填入原来的公园号码，替换框中输入其他公园号码，然后应用替换。关闭对话框后，找到文件菜单下的另存，填入合适的文件名存好。苹果系统用户的步骤也是类似的。

    注意：确保公园ID的位置标识和数字编号之间含有短横杠（-）。

    如果你忘加了，例如写成 K1234那么文件会被拒。无论对于常规的公园激活还是 P2P，都是这样。

    使用不同模式时需要不同的文件么？

    完全不用。无论是你只操作 SSB，还是 SSB/CW/AM/FT4/FT8/卫星/PSK31混着来，每个公园你都只需要一个单一的日志文件。如果你在使用 WSJT-X 时还用了其他模式，记得在上传之前将其合并到同一个 adi 文件下。

    只要你每个模式都正确进行了记录，我们的系统会自动进行处理——我们就是这么贴心！

    文件命名规则以及为什么我不能随意命名？

    在我们系统出现日志自助上传功能之前，我们的10个地区日志协调员在处理全球的日志条目——不单单是美国的！

    在那个过程中，我们发现一套合适的文件命名系统会让日志追溯、日志修复等工作变得简单。我个人会将我处理过的每一个日志存档，如果哪天有激活人说他们发现了些问题，那么我可以很快地找到这些文件。

    命名你的文件

    现在来说说命名问题：想要自助上传功能得以实现，一套标准的命名系统必须在所有开发者之间达成共识。

    这个标准是：

    callsign@park#-yyyymmdd

    我用自己的激活做个例子：

    KB2FMH@K-8088-20220202.adi

    其表示2022年2月2日在 K-8088 公园做的激活。维护一套统一的命名格式会帮上你忙的。

    如果不小心弄错了，我怎么替换那个日志文件？

    如果你日志文件有错需要处理，那么你只能联系帮助台或者你地区的协调人。用户无法自己删除日志，只能上传。

    如有错误需要处理，请在 “My Uploaded Logs” 面板该文件边上找到下拉箭头，在里面点击 “Request Help”。说清楚想要删除的文件的公园ID和日期，然后我们的支持人员在搞定后会回复通知你，这时你就可以上传新的日志来替换了。

    如果上传过程中发生问题，同样的，可以点击 “Request Help”，我们会来处理这个问题！

    最后，日志程序非常多，它们产生的各种字段也非常多。

    诸如频率、名字、网格、定位、国家等等字段在我们系统里是用不到的，所以不用去担心这些字段。

    如有问题，你可以随时联系你所在区域的协调员。

    总结

    如有本文/本文档尚未阐明的问题，可以随时联系我们的帮助台，也可以联系所在区域的协调员。凡事别慌，我们一直在这里。

    请勿偏听偏信，特别是一些所谓的专业博主。他们擅长网络营销，别有他图，他们就是专门在社交平台上干这些营生的。
