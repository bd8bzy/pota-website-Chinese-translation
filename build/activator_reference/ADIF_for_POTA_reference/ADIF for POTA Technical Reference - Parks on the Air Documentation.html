    关于 POTA 里的 ADIF 文件

    POTA 要求激活人以 ADIF 格式 提交他们的日志文件。本文对 POTA 中一些字段的使用进行了说明，同时也包含了对 ADIF 文件的一些图解示例。后者旨在进行一些 POTA 相关的描述阐释，并不是为了限制或重定义 ADIF 标准。

    ADIF 的必填字段

    想要获得有效积分，POTA 至少需要下述字段：

    STATION_CALLSIGN 或 OPERATOR

    STATION_CALLSIGN：这是电波里给出去的呼号。它可能跟 OPERATOR 一样，或者在单人激活时忽略掉。俱乐部/集体台激活就必须以俱乐部/集体台呼号填写该字段。

    OPERATOR：这是发起通联的操作员的呼号。它可能跟 STATION_CALLSIGN 一样，或者在单人激活时忽略掉。 俱乐部/集体台激活就必须包括此字段。

    每个字段只能由A到Z的字母、0到9的数字、以及斜杠 / 构成。所有其他字符包括重音符都不能出现在字段里。

    例如： <STATION_CALLSIGN:4>N8SL

    CALL

    CALL：这是猎人的呼号。其允许的字符集跟 STATION_CALLSIGN 一致。

    例如： <CALL:5>K8ERS

    QSO_DATE

    QSO_DATE：这是 QSO 开始的 UTC 日期。

    它是 YYYYMMDD 格式的8位数字，YYYY是年，MM是月，DD是日。

    不能包含未来的日期。

    月份要含两位数字，9月份之前的前缀一个0。小于01、大于12的数字会导致无效日志。例如：01表示一月，02是二月，12是十二月。日也是两位数字，九号之前的前缀一个0。例如：01是月初第一天，02是第二天，31是某月的最后一天。

    例子：2019年7月15日被记做 <QSO_DATE:8>20190715

    TIME_ON

    TIME_ON：这是 QSO 开始的 UTC 时间。 POTA 基于 QSO 的开始时间来算分，而不是结束时间。

    可以是四位或六位数字，格式分别是 HHMM 或 HHMMSS。HH是小时，从00到23；MM是分钟，从00到59；SS是秒数，从00到59。小时和分钟是必填的，秒数是可选的。

    例如： <TIME_ON:4>0945 或 <TIME_ON:6>094500

    BAND

    BAND：这是 QSO 发生的波段。

    波段是含有单位的，例如在20米上做出的 QSO，要记做 20M，而不是20。

    大部分 QSO 是单工的。如果是某些双工或者跨波段 QSO，例如卫星通联，请在本字段里填入激活人的频点，或者上行波段。

    例如： <BAND:3>20M

    MODE 或 SUBMODE

    MODE：这是 QSO 发生的模式。

    SUBMODE：这是 QSO 发生的子模式。

    ADIF 规范中有模式/子模式的相关定义。POTA 可处理的模式可参看POTA 规则的日志要求部分。 POTA 只处理一个模式字段，如果上述两个字段都填写了，POTA 会取用 submode 字段。

    例如： <MODE:3>SSB

    额外的 ADIF 字段

    一些额外的字段用于处理一些特殊的公园通联，它们被用在你的激活点位及 P2P 处理中。这些字段的主要使用场景是当激活人在同一个 UTC 日内位于多个公园，或者俱乐部/特殊事件呼号下关联有多个操作人。当 MY_SIG 及 MY_SIG_INFO 字段缺失或无效，POTA 的自助上传会提示用户去填写这些必要信息。

    MY_SIG 及 MY_SIG_INFO

    MY_SIG：特殊兴趣组。此字段填入 POTA。

    例如：

    <MY_SIG:4>POTA

    MY_SIG_INFO：指定激活人的 POTA 公园索引号码。 其格式是：POTA 活动前缀（例如K、VE等）后跟短横杠（-）然后是一个数字。

    例如：

    <MY_SIG_INFO:6>K-0005 表示美国的荒地国家公园。

    <MY_SIG_INFO:7>VE-0005 表示加拿大的班夫国家公园。

    下述额外字段会用在公园对公园通联中，以指定激活人的公园索引号码。

    SIG 及 SIG_INFO

    SIG：特殊兴趣组。这个字段填入 POTA。

    SIG_INFO：对方激活人的公园索引。

    一个跟其他激活人发起的 P2P 通联会这样来记录： <SIG:4>POTA 及 <SIG_INFO:6>K-0008

    MY_STATE

    MY_STATE：日志台站的主要行政区划编号（例如美国的州，日本的县，加拿大的省等）。如果激活的公园横跨多个行政区（州/县/省等），自助上传程序会使用该字段来定位激活人。如果缺失、无效、无法识别，程序会提示用户手动填写此信息。

    一个简单的日志文件示例

    下面这个小文件示例了一份有效格式的日志： 

    Parks On The Air Field Logger
<PROGRAMID:17>POTA Field Logger
<PROGRAMVERSION:5>0.2.8
<CREATED_TIMESTAMP:13>20201002 134537<EOH>
<STATION_CALLSIGN:5>W8MSC <CALL:5>W8TAM <QSO_DATE:8>20201002 <TIME_ON:6>134100 <BAND:3>40M <MODE:3>SSB <MY_SIG:4>POTA <MY_SIG_INFO:6>K-3315 <EOR>
<STATION_CALLSIGN:5>W8MSC <CALL:5>N3VEM <QSO_DATE:8>20201002 <TIME_ON:6>134200 <BAND:3>40M <MODE:3>SSB <MY_SIG:4>POTA <MY_SIG_INFO:6>K-3315 <EOR>
<STATION_CALLSIGN:5>W8MSC <CALL:5>W3AAX <QSO_DATE:8>20201002 <TIME_ON:6>134300 <BAND:3>40M <MODE:3>SSB <MY_SIG:4>POTA <MY_SIG_INFO:6>K-3315 <EOR>
<STATION_CALLSIGN:5>W8MSC <CALL:4>N0AW <QSO_DATE:8>20201002 <TIME_ON:6>134500 <BAND:3>40M <MODE:3>SSB <MY_SIG:4>POTA <MY_SIG_INFO:6>K-3315 <SIG:4>POTA <SIG_INFO:6>K-0008 <EOR>
<STATION_CALLSIGN:5>W8MSC <CALL:4>W5RB <QSO_DATE:8>20201002 <TIME_ON:6>134600 <BAND:3>40M <MODE:3>SSB <MY_SIG:4>POTA <MY_SIG_INFO:6>K-3315 <EOR>
<STATION_CALLSIGN:5>W8MSC <CALL:6>HI8ILO <QSO_DATE:8>20201002 <TIME_ON:6>135000 <BAND:3>20M <MODE:3>SSB <MY_SIG:4>POTA <MY_SIG_INFO:6>K-3315 <EOR>
<STATION_CALLSIGN:5>W8MSC <CALL:5>N3NWV <QSO_DATE:8>20201002 <TIME_ON:6>135100 <BAND:3>20M <MODE:3>SSB <MY_SIG:4>POTA <MY_SIG_INFO:6>K-3315 <EOR>

    本示例文件的可下载版本 在这里。

    每个有效的 QSO 都涉及激活人的台站呼号、猎人的呼号、日期、时间、波段、模式，以及激活的公园字段。每个字段之间以单个空格进行分隔。与 N0AW 相关 QSO 描述了公园对公园通联，其额外字段列出了另一个公园的信息。
